# [ADR 0043](0043-data-access-orm-strategy.md): Data Access and ORM Strategy for Node.js

## Status
Approved

## Date
2026-05-12

## Context
The ARC32 platform requires a unified, maintainable, and performant data access strategy across its progressive monolith modules. We operate in a heterogeneous database environment (primarily PostgreSQL for transactional logic, potentially MongoDB for document caching/projections). We need to select the baseline Object-Relational Mapping (ORM) and Object-Document Mapping (ODM) ecosystem that balances developer productivity (DX), enterprise features (migrations, multi-tenancy), and alignment with our Hexagonal Architecture constraints (decoupling domain from infrastructure).

## Architectural Drivers
- **Decoupling**: Strict separation between Domain Entities and Persistence Models (supporting [ADR 0002](0002-clean-architecture-nestjs.md)).
- **Multi-Tenancy**: Native or easily-integrable support for isolation strategies (Schema/Database separation, aligning with [ADR 0010](../core/0010-multi-tenancy-architecture-strategy.md)).
- **Type Safety**: End-to-end TypeScript inference to reduce runtime query errors.
- **Migration Management**: Robust CI/CD integrable lifecycle for DDL operations.
- **Performance overhead**: Minimal, with clear "escape hatches" for raw SQL when needed.

## Evaluation Criteria & Candidates

| Technology | Category | Pros | Cons | Suitability |
|------------|----------|------|------|-------------|
| **TypeORM** | RDBMS ORM | Native NestJS support, Data Mapper pattern, excellent decorator support, active enterprise ecosystem. | Can generate complex queries, Data Mapper requires discipline to avoid Active Record shortcuts. | **High** (Primary SQL) |
| **Prisma** | RDBMS ORM | Excellent DX, auto-generated client, very intuitive schema DSL. | Rust engine deployment complexity, rigid schema makes Hexagonal separation harder (generates monolithic models), weak Multi-Tenant schema switching. | Medium (Specific use cases) |
| **MikroORM** | RDBMS ORM | Unit of Work pattern, excellent clean architecture alignment, handles cyclic refs natively. | Smaller ecosystem than TypeORM, higher learning curve. | High (Alternative SQL) |
| **Knex.js** | Query Builder | Lightweight, total control over SQL, easy to performance tune. | No relationship management, manually mapping to domain objects is labor-intensive. | High (Performance/Analytics) |
| **Mongoose** | NoSQL ODM | Defacto standard for MongoDB, good schema validation, robust population logic. | Heavyweight, encourages tight coupling to Mongo schemas. | **High** (Primary NoSQL) |

## Decision

### 1. Primary Relational Persistence (RDBMS): **TypeORM**
Adopt **TypeORM** using the **Data Mapper** pattern as the enterprise standard for relational databases (PostgreSQL).
- **Justification**: Fully aligned with our Hexagonal Architecture ([ADR 0002](0002-clean-architecture-nestjs.md)). Allows distinct Entity definition in Core separate from `@Entity` Persistence Models in Infrastructure. Strongest integration with NestJS `TypeOrmModule`.
- **Constraint**: The "Active Record" pattern is strictly PROHIBITED. All access must pass through Repositories injected in the infrastructure layer.

### 2. Primary Non-Relational Persistence (NoSQL): **Mongoose**
For document-oriented use cases (aggregations, audit collections, projection read-models), **Mongoose** is the standard ODM.

### 3. Escape Hatch & High-Performance Queries: **Raw SQL / Knex**
For analytical reporting, complex JOINs causing ORM performance degradation, or bulk operations, developers are permitted to bypass the ORM using the driver-level `query()` capability or a Lightweight Query Builder (Knex), provided it stays hidden behind the Core Repository interface.

## Design Implications

### Multi-Tenancy support
TypeORM's support for connection cloning and `schema` overriding dynamically during requests aligns with our RLS/Schema-based isolation strategy ([ADR 0010](../core/0010-multi-tenancy-architecture-strategy.md)). Use `nestjs-cls` to inject tenant context into the data layer dynamically.

### Migrations
Migrations must be written explicitly via TypeScript files generated by TypeORM CLI. Automatic schema synchronization (`synchronize: true`) is FORBIDDEN in production environments.

## Consequences

### Positive
- **Consistency**: Single standard reduces onboarding cognitive load.
- **Flexibility**: Clear migration path to other drivers via Repository abstraction.
- **Maintainability**: Migrations managed in version control alongside code.

### Negative
- **Boilerplate**: Requires custom mapping code from persistence Models to Domain Entities.
- **Complexity**: Dynamic connection management required for dynamic multi-tenant routing.

## Risks and Mitigation
- **N+1 Query Problem**: Mitigation via ESLint rules for relationships and strict code review monitoring of relations fetched.
- **Vendor Lock-in**: Mitigation through strict interface adherence at Domain layer (Ports).
